# RFC 0014 Review: Codex

## Findings

1. **High: Dual-root marker precedence is not defined tightly enough to preserve RFC 0013 gates.**  
   RFC 0014 requires scripts to read both the new operations path and the legacy RFC 0013 marker path during transition (`docs/rfcs/0014-operational-artifact-home.md`, Migration Plan, lines 111-117; Acceptance Criteria, lines 139-142), but it does not say how to merge marker state across roots. RFC 0013's safety property depends on computing newest marker state per `issue_id` and `family`, requiring `ready` markers to share the same `issue_id`/`family` and explicitly supersede the older `blocked` marker, while newer `blocked` or `human_checkpoint` markers always block expansion (`docs/rfcs/0013-development-operational-issue-loop.md`, Marker schema and precedence, lines 194-198). During migration, an implementation could accidentally treat `docs/operations/.../05_REPAIR_VERIFIED.ready.md` as resolving an older `docs/reviews/.../01_RUN.blocked.md` just because it is newer, or conversely keep a legacy blocked marker active forever because cross-root `supersedes` is not specified. Proposed fix: RFC 0014 should add explicit transition precedence rules: scan both roots as one marker set, key by front-matter `issue_id` and `family`, require `supersedes` to reference the exact prior marker path even across roots, normalize paths as repo-relative strings, and state that any newest cross-root `blocked` or `human_checkpoint` remains blocking.

2. **High: The RFC blurs committed operational provenance with live runner state unless it names the `agent_runner` boundary.**  
   RFC 0014's goal is to "Separate operational run state from model review feedback" and give operational loops "one stable root" for future automation (`docs/rfcs/0014-operational-artifact-home.md`, Goals, lines 34-39). That is correct for Engram's existing marker-driven scripts, but it is unsafe as an `agent_runner` validation target unless the RFC says repo files are durable artifacts, not the authoritative control plane. The `agent_runner` spec is explicit: authoritative live state is SQLite under `.agent_runner/state.sqlite3`; repository artifacts are durable provenance only; marker files are never live control-plane state (`agent-runner/docs/SPEC.md`, Product Boundary, lines 17-19). Proposed fix: add a short "Runner boundary" note that `docs/operations/` is the committed artifact home for redacted reports/markers, while `agent_runner` validates and publishes those artifacts through its artifact table and job transitions rather than deriving queue truth from marker files. This keeps the RFC useful for validating artifact publishing and write-scope constraints without training the runner toward the old marker-as-state model.

3. **Medium: The proposed path shape drops RFC 0013's explicit `<area>/postbuild/markers/<loop_id>` structure without fully replacing its semantics.**  
   RFC 0013 defines a per-loop marker directory under `docs/reviews/<area>/postbuild/markers/<YYYYMMDD>_<run_slug>/` (`docs/rfcs/0013-development-operational-issue-loop.md`, lines 156-162). RFC 0014 proposes `docs/operations/<area>/<loop_id>/...` and gives Phase 3 as `docs/operations/phase3-postbuild/<YYYYMMDD>_<run_slug>/` (`docs/rfcs/0014-operational-artifact-home.md`, Proposal, lines 61-80). That may be a reasonable simplification, but it leaves the open question of phase-scoped versus process-scoped naming unresolved (`docs/rfcs/0014-operational-artifact-home.md`, Open Questions, lines 130-131). This is script migration risk: a later prompt cannot unambiguously map legacy `area=phase3`, `loop=postbuild`, `issue_id=<run_slug>` into the new tree or derive status output labels. Proposed fix: resolve this open question before acceptance and include a deterministic legacy-to-new mapping table.

4. **Medium: Marker schema migration is implied, not specified.**  
   RFC 0014 says committed operational artifacts inherit RFC 0013 redaction rules (`docs/rfcs/0014-operational-artifact-home.md`, Artifact Rules, lines 96-107), but it does not restate or explicitly preserve the marker front matter fields from RFC 0013 (`docs/rfcs/0013-development-operational-issue-loop.md`, lines 175-191). The migration plan says to update RFC 0013 and the runbook, but a later implementation prompt could move paths while accidentally changing `linked_report`, `supersedes`, `corpus_content_included`, or `gate` handling. Proposed fix: add an explicit rule that RFC 0013 marker front matter remains unchanged except that `linked_report` and `supersedes` may point to either `docs/operations/...` or legacy `docs/reviews/...` paths during transition.

5. **Low: The review-artifact example appears to name RFC 0013 instead of RFC 0014.**  
   RFC 0014's example review path is `docs/reviews/phase3/RFC_0013_OPERATIONAL_ARTIFACT_HOME_REVIEW_<model_slug>_2026_05_05.md` (`docs/rfcs/0014-operational-artifact-home.md`, Proposal, lines 82-86). Because the artifact under review is RFC 0014, this looks like a stale name. It is not a safety issue, but it can confuse review synthesis and runner artifact validation, where expected artifact paths are part of the work contract (`agent-runner/docs/SPEC.md`, Artifacts, lines 103-112). Proposed fix: rename the example to use `RFC_0014`.

## Runner Validation Usefulness

RFC 0014 is a good bounded validation fixture for `agent_runner` if scoped to artifact publication, review-only parallelism, unique write scopes, and verdict collection. It is not a good validation of live workflow state unless the workflow asserts that SQLite/job state remains authoritative and the repo markers are only published artifacts. That distinction matters because `agent_runner` is explicitly designed to avoid parsing terminal output or marker files as state (`agent-runner/docs/SPEC.md`, lines 17-19; `agent-runner/docs/DECISION_LOG.md`, D037, line 60).

Verdict: needs_revision
