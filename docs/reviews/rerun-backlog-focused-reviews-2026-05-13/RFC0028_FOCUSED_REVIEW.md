## Findings

### F001 - The v9 prompt artifact is not faithful to the runtime zero-claim rule

Severity: major
Source: `prompts/extraction/extractor_v9.md:112`; `src/engram/extractor.py:2260`; `src/engram/extractor.py:2288`; `docs/rfcs/0017-extraction-prompt-versioning.md:61-69`; `tests/test_phase3_claims_beliefs.py:296-309`

Rationale: RFC 0017 makes the prompt version string the join key between
persisted extraction rows and the governed prompt artifact, and says the
artifact becomes immutable once rows are persisted under that version. The new
artifact exists, but its user-prompt template says:

```text
- If no valid claims remain, return exactly {{"claims":[]}}.
```

The runtime prompt is generated by an f-string that starts at
`src/engram/extractor.py:2260`; the doubled braces at
`src/engram/extractor.py:2288` are Python escape syntax, not text sent to the
model. Rendering `build_extraction_prompt()` locally produces:

```text
- If no valid claims remain, return exactly {"claims":[]}.
```

That makes the governed artifact subtly but materially different from the
source prompt it is supposed to preserve. The new artifact test only checks for
the version string, the D082 proposed-reservation note, a system-prompt
snippet, one intent line, and `{{messages}}`, so this drift is not pinned.

Proposed fix: before any non-scratch extraction writes rows under
`extractor.v9.d082.predicate-intent`, change the artifact line to the rendered
single-brace JSON literal and add a focused artifact/provenance test that
catches this line-level mismatch. A minimal test can assert the artifact
contains `{"claims":[]}` and does not contain `{{"claims":[]}}`; a stronger
test would normalize only the intentional placeholders and compare the fixed
prompt boilerplate against `build_extraction_prompt()`.

## Resolved Checks

### B001 - Prompt artifact/provenance shape is mostly resolved, pending F001

`EXTRACTION_PROMPT_VERSION` now points at
`extractor.v9.d082.predicate-intent` (`src/engram/extractor.py:37`), and the
governed artifact now exists at `prompts/extraction/extractor_v9.md`. The
artifact records the version, runtime constant, request profile, RFC refs, and
the important D082 caveat that the decision is only a proposed reservation
(`prompts/extraction/extractor_v9.md:1-14`). `DECISION_LOG.md:106` likewise
records D082 as `proposed`, not as RFC 0028 promotion authority. The runtime
prompt also now renders predicate intent lines from the live vocabulary
metadata (`src/engram/extractor.py:83-151`, `src/engram/extractor.py:388-390`,
`src/engram/extractor.py:2251-2257`), and tests cover the artifact/version join
and prompt snippets (`tests/test_phase3_claims_beliefs.py:296-335`).

### B002 - Ambiguous subject-kind warning behavior is resolved

`subject_kind_warning()` now fetches the active entity kinds first, normalizes
them case-insensitively, and returns `None` if any active row for the subject is
`person` before consulting the curated non-person list or warning on other
entity kinds (`src/engram/interview/render.py:270-282`). That closes the prior
false-positive path for ambiguous names that legitimately have both person and
non-person entity rows. The deterministic regression at
`tests/test_interview_render.py:502-505` covers the mixed `("person",)`,
`("place",)` response and asserts no warning is emitted. The warning remains
scoped to strict person-only hints, and mixed allowed hints such as "persons or
projects" still skip the heuristic (`src/engram/interview/render.py:264-288`;
`tests/test_interview_render.py:508-523`).
