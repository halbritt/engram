## Findings

### F001 — Cited `striatum serve` precedent does not exist; D020 anchor is wrong
Severity: blocking
Source: docs/rfcs/0027-interview-web-ui.md:84-87, :118-119, :205
Rationale: The RFC anchors its localhost-only posture on `striatum serve` and cites "RFC 0012 § Local HTTP service" as precedent for "localhost-bound HTTP service inside an Engram-stack tool" (line 84-87) and again at line 118-119 and 205. RFC 0012 is `0012-python-agentic-coding-standard.md`, the Python agentic coding standard — there is no "Local HTTP service" section in it. A repo-wide search for `striatum serve` returns only this RFC. The actual D020 precedent for localhost-bound serving is **RFC 0022 / `engramd`** (docs/rfcs/0022-server-binary-api-mcp.md:178-192), which the author appears to have conflated. This matters because (a) the entire "we already do this elsewhere" argument is fictional, (b) the `--allow-non-loopback --reason` shape is being borrowed from a non-existent surface and may not match what RFC 0022 ultimately ships, and (c) the privacy posture chapter ought to cite the actual D020 line and the actual `V1_SYNTHESIS_DELTAS § Process Isolation` constraint rather than a phantom sibling. Fix the citations before this RFC reaches synthesis; the privacy claim cannot rest on a precedent that doesn't exist.

### F002 — Parallel surface to `engramd`; collides with RFC 0022 promotion path
Severity: blocking
Source: docs/rfcs/0027-interview-web-ui.md:111-116; docs/rfcs/0022-server-binary-api-mcp.md:151-159, :312-314, :386-390
Rationale: RFC 0022 explicitly states that the interview endpoints (`POST /v1/interview/sessions`, `/answer`, etc.) live in `engramd` and that the rationale for `engramd` is precisely to avoid "a parallel surface that has to be retired or merged later" (RFC 0022 lines 60-61, 312-314). RFC 0022 Open Question 9 also explicitly raises the question of whether `engramd` serves the web UI's static assets. RFC 0027 ignores both: it spins up a *separate* FastAPI app at `src/engram/interview/web.py` with its own Uvicorn lifecycle, its own bind logic, its own connection management — exactly the parallel-track problem RFC 0022 was written to prevent. Either RFC 0027 must (a) declare itself a Jinja-rendered route layer mounted onto `engramd`, (b) explicitly supersede the RFC 0022 interview endpoint list, or (c) justify why a second loopback HTTP listener on a single-user machine is preferable to one. As written, the relationship to RFC 0022 is not even mentioned in § Relationship to other artifacts; the RFC reads as if RFC 0022 doesn't exist.

### F003 — `render.py` extraction is incomplete; CLI rendering will drift
Severity: major
Source: docs/rfcs/0027-interview-web-ui.md:121-129; src/engram/cli.py:1661-1815, :1711-1725, :1928-1964
Rationale: The proposal lists four helpers to move into `src/engram/interview/render.py`: `fetch_target_display`, `fetch_evidence_excerpts`, `pick_question`, `rationale_prompt_for`. But the CLI's actual rendering surface is wider than that — `_print_evidence_excerpts` (cli.py:1711-1725) does the chronological evidence row layout, the `[idx/n] target_kind target_id stability=… conf=… …` header block (cli.py:1929-1936), the predicate-doc append (cli.py:1939-1941), the evidence-date / valid-from formatting (cli.py:1942-1961), and the `_VERDICT_VALID` / `_VERDICT_ALIAS` tables that drive `_prompt_verdict` are all rendering concerns the web layer will need too. The RFC says "no behavior change in the CLI" but the only way to get parity in question.html is to either (a) re-implement all of these in Jinja (CLI and web diverge silently the moment one is touched) or (b) extend the extraction to a fuller `interview.render` module. The "underscore-prefixed helpers" framing under-specifies the scope; a reader will land in the implementation and either re-duplicate or scope-creep. Spell out the full extraction list (header builder, evidence-row formatter, valid-from formatter, verdict-alias table) before promotion.

### F004 — htmx CDN escape clause invites D020 drift
Severity: major
Source: docs/rfcs/0027-interview-web-ui.md:155-156, :216-218; § Open question O2 (line 314-317)
Rationale: § Templates says "htmx CDN reference (or vendored copy if D020 demands)"; § Privacy posture says htmx is "either vendored to `src/engram/interview/static/htmx.min.js` or referenced by CDN; recommendation is **vendor**"; § Open Question O2 reopens the choice. The RFC explicitly claims "no outbound network" (line 218) but leaves the CDN path on the table. D020 is not a "if it demands" condition — it is an accepted decision: "Engram-reading process has no network egress; enforced at OS level." A `<script src="https://unpkg.com/htmx.org">` reference fired by the operator's *browser* is technically a different process than the serving process, but the privacy frame (HUMAN_REQUIREMENTS § "no cloud, no telemetry") makes the operator's IP and User-Agent leaking to a CDN every time they label a row obviously wrong. The RFC should remove the CDN option entirely from the v1 contract and treat O2 as closed: vendored htmx, single `<script src="/static/htmx.min.js">`, no negotiation. Leaving it open lets implementation default to CDN under deadline pressure.

### F005 — `--allow-non-loopback` escape clause is excessive for the stated use case
Severity: major
Source: docs/rfcs/0027-interview-web-ui.md:115-119, :202-209
Rationale: The RFC ships v1 with no auth, no TLS, and no CSRF (line 209: "No CSRF token in v1"). It also adds an `--allow-non-loopback --reason "..."` flag to bind to a non-loopback host. There is no use case in the RFC body that justifies a non-loopback bind — the worked example, problem statement, and out-of-scope list all assume single-user local. Adding a non-loopback escape clause to a service that has *zero* authentication and *zero* CSRF protection means the moment any operator passes `--allow-non-loopback --reason "ngrok for a demo"` they expose write-access to `gold_labels` to whatever is on the wire. The "exit 8" refusal pattern is the right default; the *flag itself* should not exist in v1. RFC 0022 (line 178-185) has the same posture but pairs non-loopback bind with a clear roadmap to token auth (RFC 0022 Open Question 7); RFC 0027 has neither auth nor a roadmap. Either drop the flag from v1 or commit to gating it behind a token-auth requirement that doesn't exist yet.

### F006 — CSRF dismissal is wrong on the threat model
Severity: major
Source: docs/rfcs/0027-interview-web-ui.md:208-209, :141-144
Rationale: § Privacy posture says "No CSRF token in v1. Localhost forms with a single user are not a CSRF threat." This is incorrect. Any web page the operator visits in any tab can issue a `POST http://127.0.0.1:8765/sessions/{id}/q/{idx}/verdict` via a hidden auto-submitting `<form>` (no CORS preflight is needed for `application/x-www-form-urlencoded` or `multipart/form-data`); the response is opaque to the attacking page but the side-effect commits a `gold_labels` row. With no CSRF token, an attacker who can guess or brute-force a `session_id` (UUID v4 — high-entropy, but the index page exposes the open-session list to anyone who can GET `/`) can poison the operator's gold-label history. The threat model is "single user" but the *browser* is a multi-tenant context where any tab can drive forms at localhost ports; this is the foundational reason `striatum serve` (and every other localhost dev surface) needs at least an `Origin`/`Sec-Fetch-Site` check or a per-session CSRF token. v1 should ship with `SameSite=Strict` cookies + an `Origin` allowlist at minimum, or a per-form token; "open for synthesis to challenge" is an admission this was not thought through.

### F007 — D044/D069 invariant is asserted but not mechanically guarded in the route surface
Severity: major
Source: docs/rfcs/0027-interview-web-ui.md:97-105, :141-144, :289-296; DECISION_LOG.md D044 / D069; docs/rfcs/0021-gold-set-interview-curation.md:512-535
Rationale: The RFC asserts "no schema layer above `gold_labels` that the CLI loop and the web UI would have to keep in sync" (line 100-101) and "No new schema beyond the persistent target order table" (line 296). But D044 is preserved in RFC 0021 by an explicit code-review invariant ("the gold-label loader must not call `engram.consolidator.transitions`") plus a schema-level guard on `belief_audit.input_claim_ids`. RFC 0027 introduces a new write path (`POST /sessions/{session_id}/q/{idx}/verdict`) that calls `agent.record_verdict`, and a new redirect target (`/sessions/{session_id}/complete` → `mark_session_completed`). Neither the route table nor the error-handling section restates the D044 invariant — i.e. nothing in the web layer says "no route may import or call `engram.consolidator.transitions`," nothing forbids a future refactor from adding a `/promote` button that *would* flip belief status. The CLI surface gets away with this because it is one file with reviewers' eyes on it; a multi-template multi-route web app multiplies the surface where a well-meaning contributor could add a "Promote belief" button. Add an explicit invariant section ("no web route may invoke any consolidator transition; no template may render a promote-belief affordance") and reference D044/D069 by name in the route table.

### F008 — `--privacy-tier-max` ceiling is described as CLI-only but `/v1/messages/{id}` returns full message body
Severity: major
Source: docs/rfcs/0027-interview-web-ui.md:142-143, :210-214, :305-306; docs/rfcs/0021-gold-set-interview-curation.md:163-167, :383-386
Rationale: The route `GET /sessions/{session_id}/messages/{message_id}` is described as "Full message body for one evidence row; htmx swap into the question page. Tier 1 enforced." But the RFC's privacy model says "Tier 1 default carry. … Higher tiers in the UI are gated by the same env-var ceiling discussion that applies to CLI rendering (open: § O3 below)" — i.e. the ceiling enforcement is specifically open-ended in v1 and the ENV var doesn't exist yet. RFC 0021 enforces the ceiling at *export*, with a hard-coded `--privacy-tier-max 1` default; the *display* path in the CLI loop currently has no tier ceiling at all (it shows whatever the operator's eyes have already seen). The web `/messages/{id}` route is a different beast: it is a programmatic surface that anything (see F006 on CSRF) can hit. Spelling out "Tier 1 enforced" in the route table without specifying *how* — and then deferring the ENV var to Open Question 3 — leaves a privacy-critical route under-specified. Resolve O3 before promotion: either commit to a hard-coded Tier 1 ceiling on `/messages/{id}` in v1, or define the ENV var explicitly with default=1, and reject requests for higher-tier rows with 403.

### F009 — Persistent target order: recommendation B underspecified; migration 011 sketch missing
Severity: major
Source: docs/rfcs/0027-interview-web-ui.md:174-198, :296
Rationale: The RFC recommends "B — `gold_label_session_targets` table" as a follow-up migration `011_gold_label_session_targets.sql` and says "no triggers needed beyond a FK to `gold_label_sessions`." But the column list is given as `(session_id, idx, target_kind, target_id, ...)` — the `...` is doing all the work. What's the type of `idx` (INT? sequence within session)? Is `(session_id, idx)` the PK? Is there a UNIQUE on `(session_id, target_kind, target_id)` to prevent dup-sample within a session? Does it carry the `candidate_pool_snapshot_id` (RFC 0021's reproducibility anchor) so a session resumed three days later still replays the same pool, or does the snapshot live only on `gold_labels` rows (i.e. the session has no anchor until the first verdict commits)? D079 / RFC 0021 § Storage holds `candidate_pool_snapshot_id` on the *label* row, not the session — Option B punctures that contract by giving the session a materialized order independent of any label. The schema sketch must address (a) how resume-after-corpus-grew is handled (does the materialized order win, or does the sampler re-run?), (b) whether the table is append-only (trigger? or insert-once-then-immutable by convention?), and (c) whether `current_gold_label` consumers need to know about it. As an "open for synthesis to challenge" line buried mid-RFC, this is too thin for an RFC that proposes a migration.

### F010 — Sampler determinism + re-sample-on-resume corner case not addressed
Severity: minor
Source: docs/rfcs/0027-interview-web-ui.md:170-198; src/engram/interview/sampler.py:1-77
Rationale: RFC 0027 says "the sampler is shared with the CLI: `GoldLabelSampler(conn, seed=...)` runs once at session creation; the sampled list is materialized into a `gold_label_session_targets` view-or-table." But the sampler is *deterministic given seed AND the candidate pool snapshot*. Between session creation and session resume (which can be days), new beliefs can land, cooldowns can expire, and `current_beliefs` can refresh. Option A ("re-sample on every request") explicitly drifts in this case; Option B ("materialize on session creation") freezes the order against a stale pool. Neither option is wrong, but the choice has eval-replay consequences (a session whose order was frozen in May 2026 cannot be replayed against the May 2026 corpus state if the corpus has moved on). The RFC should say which behavior is contractual. RFC 0021 § Sampler / § Determinism / reproducibility makes `candidate_pool_snapshot_id` load-bearing precisely for this reason; the web RFC should carry the snapshot id to the session table and document the freeze semantics.

### F011 — `--include-superseded` and `--ignore-cooldown` exposed as form checkboxes is an adversarial-sweep footgun
Severity: minor
Source: docs/rfcs/0027-interview-web-ui.md:140, :248-253; docs/rfcs/0021-gold-set-interview-curation.md:265-267, :388-390
Rationale: The index-page form is `n: [10]   seed: [4]   [ ] include superseded   [ ] ignore cooldown`. RFC 0021 frames `--include-superseded` as "for adversarial sweeps" (line 266) and notes that `--ignore-cooldown` "relaxes the cooldown filter only" but that "no flag combination relaxes the privacy tier ceiling below the default." Putting both as plain checkboxes on the new-session form (the surface a non-developer is most likely to use, per RFC 0027's stated motivation) makes adversarial-sweep mode one click away from normal operation, with no warning text, no consent banner, no audit. The CLI flag is a deliberate act; a checkbox is not. Either move these into an `[Advanced]` collapse with explanatory text, gate them behind an env var, or omit them from v1's index page (CLI-only). The RFC's point that the web UI "is the surface where ruling on 25–50 labels in a sitting becomes achievable" (line 33-34) cuts against making adversarial-sweep mode the default-visible affordance.

### F012 — No `pyproject.toml` `[serve]` extra; FastAPI / Uvicorn / Jinja2 pollute headless installs
Severity: minor
Source: docs/rfcs/0027-interview-web-ui.md:75-82, :110-116; pyproject.toml (no FastAPI/Uvicorn/Jinja2 currently declared)
Rationale: The RFC introduces FastAPI + Uvicorn + Jinja2 as new dependencies. Engram today ships with no web framework declared; adding three to the default `pip install` adds ~10 MB and a non-trivial startup cost to headless / CI / pipeline installs that never use the web UI. RFC 0027 does not address this; the obvious mechanism is a `pyproject.toml` `[project.optional-dependencies] serve = ["fastapi>=…", "uvicorn>=…", "jinja2>=…"]` block, with `engram phase3 interview serve` raising an actionable `pip install engram[serve]` error if the imports fail. The RFC should commit to optional-dependency packaging in v1; otherwise every CI run downloads the web stack to run a pipeline that never calls it.

### F013 — Test surface is hand-waved; no required-coverage list
Severity: minor
Source: docs/rfcs/0027-interview-web-ui.md:333-339 (Open Question 7)
Rationale: The test plan is "FastAPI's `TestClient` exercises routes without spinning up uvicorn; psycopg can be replaced by a test `conn` fixture. Open: do storage triggers need to be exercised from web tests?" That is the entirety of the test discussion. For an RFC that proposes new write routes touching a privacy-sensitive append-only table, the test surface should be enumerated, not deferred. Required-coverage candidates: (a) `POST /sessions/{id}/q/{idx}/verdict` happy-path round-trip; (b) trigger-rejection error banner round-trip (using a fixture row that triggers `fn_gold_labels_validate_target`); (c) 404 on unknown session_id; (d) 422 on unknown verdict; (e) `/messages/{id}` Tier 1 enforcement (a fixture message at Tier 2 must be rejected); (f) `--allow-non-loopback` refusal at startup (if the flag stays — see F005); (g) htmx is loaded from `/static/htmx.min.js`, never from a CDN URL (regression test for F004). Existing `tests/test_interview_storage.py` covers the trigger semantics; the RFC's "is the trigger fires; the route catches enough" phrasing is correct as far as it goes but should be backed by an explicit test list rather than left as Open Question 7.

### F014 — Process model open question: sync `def` vs `async def` has connection-pool implications, not just style
Severity: nit
Source: docs/rfcs/0027-interview-web-ui.md:330-333 (Open Question 6)
Rationale: O6 frames the sync-vs-async choice as "should the route signatures be `def` or `async def` from v1, even when they call `psycopg` synchronously?" That's the surface; the substance is that `async def` route handlers in FastAPI run on the event loop and *block* it on a synchronous psycopg call, which (under any concurrent load — rate limiting is also absent per line 232) head-of-line-blocks every other request. Sync `def` handlers run in a threadpool and don't have this problem. For a single-user localhost surface this rarely matters in practice, but the RFC should commit to sync `def` v1 (with the rationale "psycopg is sync; threadpool dispatch is the right primitive") rather than leave it as a coin-flip. This is a nit because the worst case is a 50ms hiccup, but the open question as written reads as cosmetic when it is operational.


## Open questions

- F005/F006 takedown: does v1 commit to `Origin`/`SameSite=Strict` enforcement, a per-session CSRF token, or both? "Single user, no CSRF" is incorrect threat-modeling; what is the actual posture?
- F008 takedown: is the `ENGRAM_GOLD_INTERVIEW_RENDER_TIER_MAX` env var part of v1 (default 1, hard-coded ceiling on `/messages/{id}`), or is the ceiling deferred? If deferred, what enforces "Tier 1 enforced" on the route in the meantime?
