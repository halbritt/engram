### A002 — `revise-rfc` is correctly grounded; `accept-rfc` and `split-rfc` were both wrong calls
Severity: minor
Source: synthesis § Recommendation
Rationale: Two blocking findings preclude `accept-rfc`: F001 is a citation fix to RFC text that has not been written, and F002 demands a § Relationship-to-RFC-0022 paragraph that does not exist in the current RFC. Both are RFC-text edits, not implementation-spec deltas, so the RFC must be revised before the spec is authored. `split-rfc` was a plausible alternative — the `render.py` extraction is independently valuable for the CLI alone, and a separable RFC could have shipped first — but the synthesis correctly judges that the migration 011 schema (which depends on `gold_label_sessions` semantics) and the render extraction are both load-bearing for the same v1 deliverable, and splitting them would create a coordination tax across two RFCs and two implementation runs for one ~200-line surface. `revise-rfc` produces a single buildable contract with one DECISION_LOG entry, one BUILD_PHASES insert, and one implementation run; the call is correct.

### A009 — Minor gap: SameSite-cookie posture from F006 ledger language is not explicitly addressed
Severity: nit
Source: synthesis O003 vs ledger F006 ("SameSite=Strict cookies plus an Origin/Sec-Fetch-Site allowlist")
Rationale: The ledger F006 wording listed "SameSite=Strict cookies plus an Origin/Sec-Fetch-Site allowlist or per-form CSRF tokens" as the v1 requirement. The synthesis adopts the Origin allowlist + Sec-Fetch-Site path and defers per-form tokens but does not explicitly comment on cookies. In v1 this is moot — there is no auth, no session cookie, no operator-identity cookie of any kind, so SameSite is not a defensible attack surface (an attacker needs *some* cookie to ride). When v1.1 introduces any cookie (including a CSRF token cookie itself), the synthesis's Origin-only posture would need to add SameSite=Strict. The synthesis's "trigger for v1.1" language ("any new mutating route added after v1") covers route additions but does not name "any new cookie introduced after v1" as a parallel trigger. The implementation spec should add cookie-introduction as a second trigger; this is a one-line edit, not a synthesis-level revision.
