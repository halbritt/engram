### F001 ŌĆö Migration filename `008_gold_labels.sql` is stale; collides with shipped 008/009
Severity: blocking
Source: `docs/rfcs/0021-gold-set-interview-curation.md:113` ("A new migration `008_gold_labels.sql` adds:"); `docs/rfcs/0021-gold-set-interview-curation.md:351` ("Land migration `008_gold_labels.sql`"); `migrations/008_claim_extractions_request_profile_unique.sql:1-10`; `migrations/009_phase4_entities_review.sql:1-209`.
Rationale: 008 already names `claim_extractions_request_profile_unique.sql` (shipped) and 009 ships the Phase 4 entities/review tables. The next available numeric prefix is `010_gold_labels.sql`. RFC must be amended at both citation sites; otherwise the build prompt will collide with `migrate()`'s checksum-tracked filename ordering. The Promotion Path step 3 also restates the wrong number, so the typo is not a single-character fix.

### F002 ŌĆö `target_id UUID` cannot foreign-key to two tables; trigger-based parent validation is the only correct shape, and the RFC omits it
Severity: blocking
Source: `docs/rfcs/0021-gold-set-interview-curation.md:117-119` (`target_kind TEXT CHECK ... target_id UUID NOT NULL`); `migrations/006_claims_beliefs.sql:131-176` (`claims.id`); `migrations/006_claims_beliefs.sql:178-237` (`beliefs.id`).
Rationale: A polymorphic `(target_kind, target_id)` cannot be enforced with a real `REFERENCES` clause in Postgres; the RFC schema declares neither a FK nor a parent-validation trigger. Without a `BEFORE INSERT` trigger that resolves `target_id` against `claims.id` or `beliefs.id` based on `target_kind`, the table will accept dangling labels ŌĆö exactly the failure mode the rest of Phase 3 protects against (compare `fn_claims_insert_prepare_validate`, `migrations/006_claims_beliefs.sql:330-470`, which validates parent rows on every insert). The RFC must either (a) split into `claim_gold_labels` / `belief_gold_labels` with real FKs, or (b) add an explicit "BEFORE INSERT trigger validates `target_id` exists in the table named by `target_kind`" clause and a contradictions-style mutation guard.

### F003 ŌĆö `target_version_stamp JSONB` is the wrong shape: claim and belief version triples are typed `TEXT` columns, JSON stamps cannot be index-joined back to source rows
Severity: major
Source: `docs/rfcs/0021-gold-set-interview-curation.md:120-124` (JSONB version stamp); `migrations/006_claims_beliefs.sql:155-157` (`claims.extraction_prompt_version TEXT`, `extraction_model_version TEXT`, `request_profile_version TEXT`); `migrations/006_claims_beliefs.sql:218-219` (`beliefs.prompt_version TEXT`, `model_version TEXT`); `migrations/006_claims_beliefs.sql:257-258` (`belief_audit.prompt_version`, `model_version`); `migrations/006_claims_beliefs.sql:169-172` (`claims_version_idx ON claims (extraction_prompt_version, extraction_model_version)`).
Rationale: Both claims and beliefs store version triples as plain `TEXT` columns and there is a btree index on the claim version pair. A JSONB stamp on `gold_labels` cannot use that index when the gold-set runner asks "all labels for `extractor.v5.d063.validation-repair-audited`"; every query becomes a JSONB extract + cast. Promote to typed columns: `extraction_prompt_version TEXT`, `extraction_model_version TEXT`, `request_profile_version TEXT`, plus (for `target_kind='belief'`) `consolidator_prompt_version`/`consolidator_model_version`. Belief versions can be NULL when target_kind='claim' (and vice versa) under a CHECK. Note also that the RFC body says belief stamp uses "(consolidation_prompt_version, consolidation_model_version, request_profile_version) ŌĆö equivalents already stored on belief_audit"; but `beliefs` itself stores only `prompt_version` and `model_version`, and `belief_audit` (`migrations/006_claims_beliefs.sql:239-264`) does not define a `request_profile_version` column at all. The RFC's column citation is incorrect.

### F004 ŌĆö Append-only is asserted but no Postgres-level enforcement is specified, breaking parity with raw evidence and `claim_audits`
Severity: major
Source: `docs/rfcs/0021-gold-set-interview-curation.md:139-140` ("Append-only. No `UPDATE`/`DELETE` enforcement matches the raw-evidence discipline"); `migrations/006_claims_beliefs.sql:472-486` (`fn_claims_insert_only` + `claims_insert_only` trigger); `migrations/006_claims_beliefs.sql:595-609` (`fn_belief_audit_append_only`); `migrations/007_claim_audits.sql:104-118` (`fn_claim_audits_append_only`).
Rationale: Every other "append-only" table in this codebase enforces the rule with a `BEFORE UPDATE OR DELETE` trigger that raises `P0001`. The RFC's prose phrase "no `UPDATE`/`DELETE` enforcement matches the raw-evidence discipline" is ambiguous ŌĆö read literally, it could mean "no enforcement, matching the (non-)discipline of raw evidence" rather than "we will write a trigger like the rest." The RFC must explicitly require `fn_gold_labels_append_only` mirroring `fn_claim_audits_append_only`. Otherwise an implementer following the literal text ships a table with no guard.

### F005 ŌĆö Sampler determinism claim is under-specified: seed alone does not yield replay without freezing the input row set and active-learning bias inputs
Severity: major
Source: `docs/rfcs/0021-gold-set-interview-curation.md:165-167` ("The sampler is seeded; the sampler ID + version + seed + strata weights are stamped on each emitted question so an interview pass is reproducible.").
Rationale: Reproducibility requires (a) a seed, (b) a deterministic ordering of the candidate set drawn from the corpus at sample time, and (c) snapshotting any active-learning signal feeding the bias (e.g., RFC 0018 reviewer scores, which are themselves time-varying). The RFC stamps seed + strata weights but does not specify (b) or (c). New claims/beliefs landing between seed-A and seed-B replays will silently change the candidate pool and produce a different question sequence under the same seed. To replay, the sampler must also stamp either `corpus_snapshot_id` or per-question `(target_kind, target_id, target_version_stamp)` of the candidate pool. Without that, "reproducible" is aspirational. Recommend either freezing on session creation (e.g., a materialized `gold_label_session_candidates` row set) or dropping the determinism claim from V1 prose.

### F006 ŌĆö `sampler_strata_key JSONB` is the wrong shape for sampler determinism; the strata schema should be a typed lookup, not free-form JSON
Severity: major
Source: `docs/rfcs/0021-gold-set-interview-curation.md:131-133` (`sampler_strata_key JSONB NOT NULL  -- e.g. {"stability_class":"goal","conf_band":"0.6-0.8"}`).
Rationale: A free-form JSONB strata key is unindexable for the sampler's "under-labeled strata" weighting query, which is the sampler's core hot loop. It also lets a future code change silently introduce a new key (`recency_band`, `belief_status`) without breaking existing rows ŌĆö a new bias becomes invisible against historical labels. Either (a) define a `sampler_strata_definition` lookup table (analog to `predicate_vocabulary`, `migrations/006_claims_beliefs.sql:37-62`) with a `version` column and FK from `gold_labels`, or (b) split into typed columns `stability_class TEXT`, `confidence_band TEXT`, `recency_band TEXT`, `belief_status TEXT NULL`, with checks. Option (b) matches the existing codebase pattern.

### F007 ŌĆö CLI surface mismatch: argparse uses phase-scoped subparsers + deprecated bare commands; RFC bare `engram interview` does not slot into either pattern
Severity: major
Source: `docs/rfcs/0021-gold-set-interview-curation.md:194-201` (`engram interview start | resume | history | export`); `src/engram/cli.py:74-75` (root parser); `src/engram/cli.py:281-419` (phase-scoped subparsers `phase1`/`phase2`/`phase3`/`phase4`); `src/engram/cli.py:64-70` (`warn_legacy_command` + `legacy_replacement`).
Rationale: Master just landed RFC 0025's command surface: every new command must either be phase-scoped (`engram phaseN <verb>`) or land as a bare deprecation alias to a phase-scoped form. `engram interview` is neither ŌĆö it is a top-level bare command with its own subparser tree. The RFC needs to declare a phase placement before it lands. Two viable shapes: (a) `engram phase3 interview {start,resume,history,export}` (since labels join Phase 3 IDs) ŌĆö preferred per the existing pattern; or (b) a new top-level `interview` parser explicitly carved out as "cross-phase substrate," in which case BUILD_PHASES needs a row for the gold-set substrate. The RFC must pick one before the build prompt can land; the current name slot is also missing from `cli.py`'s subparser registry.

### F008 ŌĆö `prompt_template_id` + `prompt_template_version` divergence from RFC 0017 versioning convention
Severity: major
Source: `docs/rfcs/0021-gold-set-interview-curation.md:126-127` (separate `prompt_template_id TEXT`, `prompt_template_version TEXT`); `docs/rfcs/0021-gold-set-interview-curation.md:188-190` ("Templates live under `prompts/interview/` and carry `prompt_template_id` + `prompt_template_version`"); `docs/rfcs/0017-extraction-prompt-versioning.md:64-75` (format `extractor.v{N}.{date_or_decision}.{descriptor}`, artifact path `prompts/extraction/extractor_v{N}.md`, "immutable once a row is persisted under that version").
Rationale: RFC 0017's contract is a single composite version string (`{area}.v{N}.{date_or_decision}.{descriptor}`) joined to a single artifact path. The RFC 0021 schema splits this into two columns (`prompt_template_id` + `prompt_template_version`), which (a) implies the id is independent of the version (`interview.evidence-trace` + `v3`), and (b) breaks the join key pattern that `claims.extraction_prompt_version` already uses (single string). To follow RFC 0017, the column should be one `prompt_template_version TEXT` like `interview.v1.d044.belief-currentness`, and the artifact lives at `prompts/interview/interview_v1.md`. If the RFC really wants id + version split, it must justify the divergence and add a CHECK that `prompt_template_id` is the prefix of `prompt_template_version` to keep the join intact.

### F009 ŌĆö No session/run identity column: `gold_label_session` is asked for in the prompt's review checklist but the RFC neither provides it nor uses it
Severity: major
Source: `docs/rfcs/0021-gold-set-interview-curation.md:115-137` (full schema, no `session_id` / `run_id` / `sampler_seed` column); `docs/rfcs/0021-gold-set-interview-curation.md:218-254` (worked example shows `session: gl-sess-2026-05-07-00`); `docs/rfcs/0021-gold-set-interview-curation.md:194-198` (`engram interview resume [--session-id <id>]`).
Rationale: The CLI exposes `--session-id`, the worked example prints `session: gl-sess-2026-05-07-00`, and the sampler determinism claim depends on a stable seed-per-session ŌĆö but `gold_labels` has no `session_id` or `seed` column, only `sampler_id` + `sampler_version`. Two related issues: resume cannot be implemented without recovering the seed and remaining-question pool, and "verdicts within a session" is unqueryable. Add either a `gold_label_session` parent table (id, sampler_id, sampler_version, seed, strata_weights JSONB, started_at, closed_at) with FK on `gold_labels.session_id`, or denormalize `(session_id, sampler_seed)` onto `gold_labels`. The parent-table form matches the `claim_extractions` ŌåÆ `claims` pattern (`migrations/006_claims_beliefs.sql:104-118` parent + child).

### F010 ŌĆö `privacy_tier INT` carry has no enforcement: nothing copies the target row's tier on insert, and export `--privacy-tier-max` defaults are unspecified
Severity: major
Source: `docs/rfcs/0021-gold-set-interview-curation.md:136` (`privacy_tier INT NOT NULL    -- carried from target row`); `docs/rfcs/0021-gold-set-interview-curation.md:200` (`engram interview export --format jsonl [--privacy-tier-max <N>]`); `migrations/006_claims_beliefs.sql:158` (`claims.privacy_tier INT NOT NULL`); `migrations/006_claims_beliefs.sql:220` (`beliefs.privacy_tier INT NOT NULL`).
Rationale: "Carried from target row" is enforced nowhere. A bug in the interview agent that defaults `privacy_tier=0` will silently downgrade a Tier 2 belief's label, and export with no `--privacy-tier-max` ceiling will leak it. Two needed changes: (1) a `BEFORE INSERT` trigger on `gold_labels` that joins to the parent (claim or belief) and either copies or asserts equality on `privacy_tier`; (2) a default for `--privacy-tier-max` (the prompt suggests "matches the user's working tier" ŌĆö the RFC needs an explicit env var or config value, otherwise the default is "no ceiling"). Compare `claims.privacy_tier` which is set explicitly by extractor code rather than enforced by trigger ŌĆö gold labels should be stricter, not looser, since they are the export surface.

### F011 ŌĆö `verdict='unsupported'` overlaps `claim_audits.verdict='invalidated'` and is not differentiated from `false`; vocabulary is a vocabulary-table candidate, not a CHECK enum
Severity: minor
Source: `docs/rfcs/0021-gold-set-interview-curation.md:128-129` (`verdict CHECK ... ('true','false','stale','unsupported','unsure','skip')`); `migrations/007_claim_audits.sql:41-44` (`claim_audits.verdict IN ('supported','partial','invalidated')`); `docs/rfcs/0018-evidence-to-claim-audit-cascade.md:170-201` (Stage 2 verdict semantics).
Rationale: `unsupported` looks like a copy of RFC 0018's `invalidated` semantics ("evidence does not establish the claim"), but the RFC does not distinguish: is `false` "the claim is wrong as a matter of fact in the user's life," and `unsupported` "the user can't recall whether it's true but the cited evidence doesn't show it"? D069 says gold labels and audit cascade are independent producers; without a definitional split between `false` and `unsupported`, downstream consumers cannot tell whether a `false` verdict is a fact-correction signal or an evidence-trace signal. Two recommendations: (a) document the distinction in the RFC body (add a one-line gloss per verdict), and (b) put the verdict vocabulary in a `gold_label_verdict_vocabulary` table with `description` + `corresponds_to_audit_reason` columns, mirroring `audit_reason_vocabulary` (`migrations/007_claim_audits.sql:10-35`), so the join from gold labels to audit reasons is structural rather than implicit.

### F012 ŌĆö Active-learning bias "wired but defaulted off" has no schema affordance: there is no `active_learning_signal_id` or feature-store column on `gold_labels`
Severity: minor
Source: `docs/rfcs/0021-gold-set-interview-curation.md:158-171` (active-learning bias section); `docs/rfcs/0021-gold-set-interview-curation.md:152-167` (sampler section).
Rationale: The RFC says active-learning bias is wired but off in V1, with RFC 0018 reviewer scores as the eventual feed. But the schema has no place to record which bias signal informed a given pick ŌĆö `sampler_strata_key` is JSONB-free-form, and there's no `bias_signal_version` column. When the bias is enabled, replays under the same seed will diverge silently. Either (a) reserve a column `active_learning_signal_version TEXT NULL` and stamp it on every emitted row even in V1 (NULL when bias is off), or (b) state explicitly in the RFC that turning active-learning on constitutes a sampler version bump (so `sampler_version='stratified.v2.with-aris-bias'`). Option (b) is cleaner and consistent with RFC 0017's bump rules.

### F013 ŌĆö Worked example is internally inconsistent with the schema: predicate `is_related_to` uses `kind` but vocabulary requires `name` and `kind`
Severity: nit
Source: `docs/rfcs/0021-gold-set-interview-curation.md:247-250` (`belief b-aa90... "user is_related_to {name:'Sam',kind:'sibling'}"`); `migrations/006_claims_beliefs.sql:80` (`('is_related_to', 'relationship', 'multi_current', 'json', '{name}', '{name,kind}', 'family relation')`).
Rationale: The example checks out (both `name` and `kind` present), but the prior example `[2/5] claim c-91d2... predicate=has_pet, object={"name":"Mochi","species":"cat"}` matches the vocabulary. Less an outright error than a weak axis: the worked example should also exercise the contradiction-mode pilot question (RFC ┬¦ Open questions item 3) since "deferred to v1.5" still informs the schema decisions today. As is, the example only exercises the trivial single-row interview path and does not show how a `gold_labels` row would record a contradiction-question verdict ŌĆö which the schema currently has no shape to express (a contradiction question implies two `target_id`s).

### F014 ŌĆö Cooldown defaults are a placeholder; no schema column nor enforcement is specified
Severity: minor
Source: `docs/rfcs/0021-gold-set-interview-curation.md:163-164` ("A target answered in the last N days is suppressed for that window"); `docs/rfcs/0021-gold-set-interview-curation.md:339-341` (Open question 6 defers values).
Rationale: Cooldown is sampler-side only, but its inputs come from `gold_labels` (most-recent `answered_at` per target). Without a `current_gold_label` view (the RFC mentions one casually at line 256 but does not define it), every sampler invocation re-scans `gold_labels`. Add the view definition explicitly to the RFC schema section, mirroring `current_beliefs` in `migrations/009_phase4_entities_review.sql:143-181`. The cooldown N is correctly deferred to empirical tuning, but the view it depends on is not optional.

### F015 ŌĆö Re-asks producing new rows breaks the unique-question-per-target invariant the sampler implicitly assumes
Severity: minor
Source: `docs/rfcs/0021-gold-set-interview-curation.md:139-141` ("later verdicts on the same target produce new rows"); `docs/rfcs/0021-gold-set-interview-curation.md:160-161` ("targets with no prior `gold_labels` row at the current `target_version_stamp`").
Rationale: The cooldown bias says "no prior label at current version stamp" ŌĆö but if the user re-asks (skipping cooldown), two rows now exist for the same `(target_id, target_version_stamp)`. The sampler's "no prior label" check has to read `current_gold_label` (latest per target+version), not raw `gold_labels`. The RFC implies this but does not say it. Add: "The sampler's bias query uses `current_gold_label` (most recent verdict per `(target_kind, target_id, target_version_stamp)`); raw `gold_labels` is the audit trail."

## Open questions

- Should `gold_labels` reference `claim_extractions.id` (which is a real FK target with version triple guaranteed) rather than `claims.id`, so version-stamp consistency is structural?
- If `engram interview` is phase-scoped, does it land under Phase 3 or in a new "Step 5 substrate" phase row in BUILD_PHASES ŌĆö i.e., is gold-set authoring a Phase 3 follow-on or a cross-phase eval substrate?
- Does the contradictions-mode question (RFC ┬¦ Open Q 3) need its own table from day one (`gold_labels` with two target_ids breaks the polymorphic shape further), or can v1.5 retrofit cleanly?
- Should `gold_labels` (or its parent session table) carry an `engram_corpus_snapshot_id` so replays can pin the candidate pool, even if the snapshot itself is materialized lazily?
- Is the export's privacy-tier ceiling default `ENGRAM_WORKING_PRIVACY_TIER` (an env var that does not yet exist), or should the RFC define a new env var for the gold-set surface specifically?
- Should the agent capture the rendered prompt's hash (e.g., SHA256 of `prompt_text`) so later prompt-template churn does not silently re-render the historical question into something the user didn't actually answer?
- Does `belief_audit.evidence_message_ids` (`migrations/006_claims_beliefs.sql:260`) supersede `belief_audit.evidence_episode_ids` referenced in RFC 0011 line 243 ŌĆö i.e., does the RFC need to be updated to reflect the shipped column name when it stamps belief versions?
