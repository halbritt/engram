# RFC 0046: Engram Striatum Projection And Index Schema

| Field | Value |

| RFC | RFC-0046 |

| Implementation | none |

## Summary

This RFC is the scaffold for rebuildable Engram projections over a Striatum
corpus. Raw Striatum evidence should remain append-only; derived tables and
indexes should be rebuildable from that evidence and the accepted corpus
contract.

This scaffold does not choose the final schema. It queues the design and review
work needed after RFC 0045 defines the export contract.

## Roadmap Position

RFC 0046 follows RFC 0045. It turns a validated Striatum bundle into structured
query surfaces for runs, workflows, agents, artifacts, RFCs, designs, reviews,
syntheses, commits, issues, blockers, reports, changelogs, and cross-links.

## Goals

1. Define projection tables for the Striatum corpus without making raw evidence
   mutable.
2. Define rebuild rules, derivation versions, and provenance references.
3. Define indexes needed for identifier lookup, structured filtering, and
   later layered retrieval.
4. Preserve tenant and corpus isolation from RFC 0044 and RFC 0045.
5. Keep projections useful for future local application memories beyond
   Striatum.

## Non-Goals

- No Striatum exporter implementation.
- No context-injection policy.
- No MCP tool expansion unless a later RFC accepts it.
- No claim/belief creation from Striatum data in this pass.
- No personal-memory projection redesign.

## Dependencies

- RFC 0045 Striatum Corpus Contract V2.
- RFC 0044 hardening cleanup for tenant/source-kind consistency.
- Current Engram schema documentation and migration discipline.

## Scaffolded Workstreams

1. Map V2 bundle source kinds to candidate projection entities.
2. Identify cross-link fields and stable references required for auditability.
3. Draft migration and rebuild semantics.
4. Draft indexing requirements for exact IDs, file paths, run IDs, commit SHAs,
   issues, RFCs, and artifact references.
5. Define projection freshness and stale-index detection hooks for RFC 0049.

## Review Requirements

Use the multi-agent review loop before promotion:

- schema/migration safety review;
- provenance and rebuildability review;
- tenant/corpus isolation review;
- query ergonomics review.

## Acceptance Criteria To Define

- Projection rows can be rebuilt from raw Striatum evidence.
- Every projection row cites raw evidence and derivation version.
- Tenant/corpus boundaries are structurally preserved.
- Indexes support exact reference lookup and structured filters.
- Schema docs are generated by the accepted project mechanism, not hand-edited.

## Open Questions

1. Which projections should be first-class tables versus JSON metadata?
2. Which cross-links must be materialized, and which can be computed at query
   time?
3. How should projection rebuild state be recorded?
4. How should future non-Striatum application memories share or avoid these
   tables?
