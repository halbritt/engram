# RFC 0004: Critique-To-Action Loop

Status: proposed
Date: 2026-05-06
Context: ARIS paper review, `agent-runner/docs/SPEC.md`,
`docs/process/multi-agent-review-loop.md`

## Problem

The current review loop captures verdicts and review artifacts, then a synthesis
classifies findings. That is enough for RFC disposition, but it is weaker for
implementation and repair workflows where the executor must prove that concrete
review items were addressed.

Without stable action-item ids, a workflow can drift from "reviewer requested
X" to "executor says the revision handled the review" without a direct
resolution ledger.

## Goals

- Normalize actionable review feedback into stable action-item ids.
- Require revision jobs to mark each action item as resolved, modified,
  deferred, or rejected.
- Let re-review jobs verify action-item resolution directly.
- Preserve the existing findings-ledger and synthesis model for RFC workflows.

## Non-Goals

- Do not require action-item ledgers for every review.
- Do not let reviewers patch source artifacts directly unless assigned.
- Do not replace human checkpoints for contested findings.

## Proposal

Add a recommended artifact kind:

```text
action_item_ledger
```

An action-item ledger records:

- stable action id;
- source review artifact and finding id;
- requested change;
- owner job;
- resolution status;
- resolution evidence;
- notes for re-review.

Resolution statuses:

- `resolved`
- `resolved_with_modification`
- `deferred`
- `rejected`
- `blocked`

Recommended workflow pattern for implementation reviews:

```text
review -> action_item_ledger -> revision -> re-review
```

The revision job updates or emits the ledger. The re-review job reads the
ledger and verifies each row against the changed artifacts.

## Acceptance Criteria

- Workflow validation accepts `action_item_ledger` as an artifact kind.
- Example prompts exist for action-item ledger creation and re-review.
- Re-review work packets include source review artifacts and the
  action-item ledger.
- The runner evidence export includes action-item ledgers as curated artifacts.

## Open Questions

- Should action-item ids be generated by the ledger job only, or may individual
  reviewers emit proposed ids?
- Should unresolved high-severity action items automatically force
  `needs_revision`, or should that remain prompt-level policy in V1?
